# CONTRIBUTING

開発の原則と禁止事項。

**実行手順は `.claude/skills/` に分離されている。** ここでは「何を守るか」「なぜそうするか」だけを記述する。

> 実装規律の4原則 (DRY / YAGNI / TDD / Frequent commits) は [CLAUDE.md](./CLAUDE.md) を参照。

---

## 原則

### 1. GitHub Flow

- `main` は常にデプロイ可能な状態を保つ
- 全ての変更は feature ブランチ + PR 経由
- main への直接 commit / push は禁止

### 2. ブランチ命名

```
<type>/<issue#>-<short-summary>
```

| type | 用途 | 例 |
|------|------|-----|
| `feature/` | 新機能 | `feature/42-user-profile` |
| `bugfix/` | バグ修正 | `bugfix/57-login-error` |
| `hotfix/` | 緊急修正 (本番障害) | `hotfix/61-payment-down` |
| `refactor/` | リファクタ | `refactor/70-extract-service` |
| `docs/` | ドキュメント変更 | `docs/72-update-readme` |

### 3. Conventional Commits

```
<type>(<scope>): <subject> (issue#<番号>)
```

| type | 用途 |
|------|------|
| `feat` | 新機能 |
| `fix` | バグ修正 |
| `docs` | ドキュメント |
| `refactor` | リファクタ (機能変更なし) |
| `test` | テスト追加・修正 |
| `chore` | ビルド・ツール・依存関係 |
| `perf` | パフォーマンス改善 |
| `style` | コードスタイル (機能影響なし) |

### 4. Squash and Merge + Delete Branch

- マージ戦略は **Squash and Merge** に統一 (main の履歴を 1 PR = 1 commit に保つ)
- マージ後はブランチを即削除

---

## 禁止事項

### AI 生成メッセージの記載

コミットメッセージ・PR 本文・Issue 本文に、AI が生成した旨を記載しない。

禁止例:
- `🤖 Generated with Claude Code`
- `Co-Authored-By: Claude Sonnet <noreply@anthropic.com>`
- `Generated by GitHub Copilot`

理由: 履歴は変更内容を記録するもので、使用ツールを宣伝する場所ではない。

### 破壊的 git 操作

明示許可なく以下を実行しない:
- `git push --force` (`-f`)
- `git reset --hard`
- `git branch -D`
- `git clean -fd`
- 管理下ファイルの `rm -rf`

### 推測 commit / PR

「動くはず」「テスト通るはず」での commit / PR は禁止。実行確認の上で進める。

## 実行手順 (workflow)

具体的な手順は skill に集約されている。

| やりたいこと | skill |
|------------|------|
| Issue を作る (工数見積込み) | `issue-create` |
| 設計を詰める / 用語整理 | `grill-with-docs` |
| テスト駆動で実装 | `tdd` |
| バグ調査 | `diagnose` |
| 構造リファクタ | `improve-architecture` |
| コミット | `commit` |
| PR 出してマージしてブランチ削除 | `finishing-branch` |

詳細は `.claude/skills/<name>/SKILL.md` を参照。
