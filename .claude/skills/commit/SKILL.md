# commit

変更を Conventional Commits 形式で commit する。AI footer は付けない。

## Pre-flight checks

実行前に必ず確認:

1. **main ブランチで作業していないか**

   ```bash
   git branch --show-current
   ```

   `main` だったら **即停止**。「main で作業しています。`feature/<#>-<summary>` ブランチを作成しますか?」とユーザーに確認する。

2. **staged changes があるか**

   ```bash
   git status --short
   ```

   何も無ければ、ユーザーに何を commit したいか確認し、必要なら `git add <files>` する。`git add -A` や `git add .` は秘密ファイル混入リスクがあるので **使わない**。ファイルを個別指定。

## 1. Issue 番号の抽出

現在のブランチ名から自動抽出:

```bash
git branch --show-current | grep -oE '^[a-z]+/[0-9]+' | grep -oE '[0-9]+$'
```

抽出パターン: `feature/42-...`、`bugfix/57-...`、`hotfix/61-...`、`refactor/70-...`、`docs/72-...` から番号部分。

抽出できない場合 (ブランチ名が規約から外れている等) は、ユーザーに番号を確認するか、`(issue#)` を省略する。

## 2. 差分から commit メッセージ案を作る

```bash
git diff --cached --stat
git diff --cached
```

差分を見て:

- **type** を選定: feat / fix / docs / refactor / test / chore / perf / style
- **scope** (任意): 影響領域 (`auth`、`api`、`db` 等)。曖昧なら省略
- **subject**: 50文字以内、命令形、現在形 (「〜を追加」「〜を修正」)

提案フォーマット:

```
<type>(<scope>): <subject> (issue#<N>)
```

例:

```
feat(auth): ログイン機能を追加 (issue#42)
fix(cart): 合計金額の計算を修正 (issue#57)
docs: CONTRIBUTING を更新 (issue#1)
refactor(user): サービスオブジェクトに抽出 (issue#45)
```

## 3. ユーザーに提示・確認

メッセージ案をユーザーに見せて承認を取る。修正があれば反映してから commit。

## 4. Commit 実行

```bash
git commit -m "<type>(<scope>): <subject> (issue#<N>)"
```

**禁止事項 — これらは絶対に付けない:**

```
🤖 Generated with Claude Code               ❌
Co-Authored-By: Claude <noreply@...>        ❌
Co-Authored-By: Claude Sonnet/Opus ...      ❌
Generated by GitHub Copilot                 ❌
```

複数行メッセージが必要な場合 (本文 / footer) でも、AI 関連の文言は一切混入させない。

```bash
git commit -m "$(cat <<'EOF'
feat(auth): ログイン機能を追加 (issue#42)

セッション管理と二段階認証を含む。
EOF
)"
```

## 5. 完了報告

```bash
git log -1 --oneline
```

直近 commit を表示してユーザーに完了を伝える。

## エッジケース

- **複数の論理的変更が staged されている**: 「これは2つの変更が混ざっています。分けますか?」と確認。`git reset` で staging を解除し、ファイル単位で add し直す手順を提示
- **subject が 50 文字超**: 簡潔化を試みる。どうしても長いなら本文に詳細を回し subject は要約
- **issue# が無い (規約外ブランチ)**: ユーザーに番号を聞くか、`(issue#)` を省略する判断をユーザーに委ねる
